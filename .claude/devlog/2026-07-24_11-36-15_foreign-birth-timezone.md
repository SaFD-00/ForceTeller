## 발견

지역 입력을 프론트 입력단 → API → 만세력 경도 보정까지 전 경로로 실측했다.
도시 검색은 geonamescache 기반 **전세계** 도시이고 UI 폴백 목록에도 뉴욕·런던·
시드니가 1급으로 노출되는데, 엔진은 입력 시각을 한국 벽시계로 전제한 채
`(현지경도 − 한국 표준자오선)×4분`을 보정하고 있었다. 실측: 뉴욕 현지 14:30
출생 → −836분(−13.9h) 보정 → "진태양시 00:37" → 시주가 자시(병자)로 오판.
자정 부근이면 일주까지 어긋난다. 부수 결함으로, 프론트가 도시 이름만 보내고
백엔드가 이름으로 재조회하므로 동명(同名) 도시(런던 UK vs CA 등)가 인구순으로
오해석될 수 있었다.

## 수정

출생 도시의 IANA 시간대(geonames `timezone` 필드, 또는 API/CLI `timezone` 명시
입력)로 **현지 시각 → 한국 벽시계 환산 후 기존 파이프라인 통과**. 핵심 근거:
환산 후의 기존 보정 `(현지경도 − 135°)×4분`은 전개하면 현지 시태양시와
수학적으로 동일하다(`경도×4 − 오프셋×60`으로 양변 일치). 따라서 시주·일주는
현지 태양시 기준으로 서고, 절기 판정은 절대 동시성(KST 절입 시각) 기준으로
성립한다 — 별도 유파 선택을 강요하지 않는 최소 수정.

- 한국(Asia/Seoul)이면 무변환 — 1954~1961 UTC+8:30 등 역사 규약은 KSTHistory 영역 유지
- `--longitude` 단독 지정은 city 기본값 Seoul → Asia/Seoul 조회로 자연히 무변환(KST 입력 규약 유지)
- 환산은 진태양시 유파 선택(`apply_time_correction`)과 무관하게 적용 (타임라인 정합 문제)
- `TimeCorrection`에 `birth_timezone`·`korean_time` 노출, 미지 시간대는 400
- 프론트는 선택 도시의 `longitude`·`timezone`을 명시 전송 — 동명 도시 오해석 차단, 수동 수정 시 잔류 해제
